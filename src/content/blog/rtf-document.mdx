## RTF File Specification

To better understand the RTF file format, I highly recommend reading the following documentation.

&#x2192; <a href="https://www.biblioscape.com/rtf15_spec.htm" target="_blank">Rich Text Format (RTF) Version 1.5 Specification</a>

Here, I write a simple RTF file called `example.rtf`, which displays the text **“Hwangstice - On the Cloud.”**

![image.png](/image/rtf-document/image.png)

This is the code behind the scenes.

```
{\rtf1\ansi\deff0 {\fonttbl {\f0 Courier;}}{\colortbl;\red0\green0\blue0;\red255\green0\blue0;\red0\green255\blue0;\red0\green0\blue255;}\cf2 \b \qc \fs32 Hwangstice \cf0 \b - \cf4 \b On the Cloud.\line}
```

The logic of RTF is pretty straightforward. An RTF document is enclosed within a pair of curly braces `{}` that define the root group of the document. It begins with the control word `\rtf1`, which identifies the file as an RTF document, followed by a series of other control words such as `\ansi`, `\deff0`, and many more.

At this point, you might be wondering what a control word is. To be simple, a control word is like a **"command"** that tells the RTF reader how to format or interpret the document. Every control word starts with a backslash (`\`) and may optionally contain a numeric parameter.

Control words can be organized into groups using curly braces `{}`. For example, the `\fonttbl` and `\colortbl` control words shown above define the font table and color table used by the document.

A control word ends when the parser encounters a non-alphabetic character, such as a space, a backslash (`\`) that starts another control word, or a brace (`{`). If a space is used to terminate a control word, that space is usually consumed by the parser and does not appear in the final document.

To include a literal backslash character in the document text, use `\\`.

## What the Shell?

Yes, this is the key question you should ask yourself when analyzing an RTF file.

RTF files are well known for supporting embedded objects through the control word `\object`. This is typically where the shellcode or exploit-related data is stored within the document.

According to the <a href="https://www.biblioscape.com/rtf15_spec.htm" target="_blank">RTF file documentation</a>, this is the structure you will usually encounter when viewing the raw contents of an RTF file.

![](/image/rtf-document/image%201.png)

<p align="center">
  <em>Source: <a href="https://www.biblioscape.com/rtf15_spec.htm#Heading50" target="_blank">RTF Documentaion - Objects</a></em>
</p>

In RTF, an object consists of two parts:

- Data part
- Result part

The data part is hidden from the document and contains the actual object data. The result part is used to render the object's appearance when the document is displayed or when other applications use the embedded or linked object.

Here, I will use the Proof of Concept (PoC) of **`CVE-2017-11882`** from <a href="https://github.com/embedi/CVE-2017-11882/blob/master/example/exploit.rtf" target="_blank">here</a> to demonstrate things more clearly.

![](/image/rtf-document/image%202.png)

<p align="center">
  <em>Source: <a href="https://github.com/embedi/CVE-2017-11882/blob/master/example/exploit.rtf" target="_blank">PoC - exploit.rtf</a></em>
</p>

In the image above, the data part is highlighted by the red rectangle, while the result part is highlighted by the green rectangle. For now, let's focus on the data part.

The fields of interest here are `\objemb`, `\objupdate`, and `\objdata`.

- `\objemb` indicates that an OLE object is embedded within the document.
- `\objupdate` forces the application to update the object before displaying it.
- `\objdata` contains a hexadecimal blob in little-endian format that stores the object's data.

Normally, when you encounter a blob of hexadecimal data, it must follow a specific format, right?

This example is not an exception. Here, the object data is stored in the **OleSaveToStream** format. This format begins with an <a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-oleds/3c695db0-943f-48a0-b134-939af3b3a4ca?redirectedfrom=MSDN" target="_blank">ObjectHeader</a>, as shown below.

![](/image/rtf-document/70c22477-9eca-4104-8ab1-a64c9f16aa82.png)

<p align="center">
  <em>Source: <a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-oleds/3c695db0-943f-48a0-b134-939af3b3a4ca?redirectedfrom=MSDN" target="_blank">[MS-OLEDS]: ObjectHeader</a></em>
</p>

The first four bytes represent the **OLEVersion**, which is described in the specification as **"set to any arbitrary value and MUST be ignored on receipt"**. Therefore, we can safely ignore these bytes.

The next four bytes represent the **FormatID**. According to the specification, a value of `0x00000001` indicates a **LinkedObject** structure, while a value of `0x00000002` indicates an **EmbeddedObject** structure. As shown in the following image, our **FormatID** is set to `0x00000002`, telling us that the data contains an **EmbeddedObject** structure.

![image.png](/image/rtf-document/c021f6a8-4b76-4502-949f-4db9aac3f935.png)

The next field is the **ClassName**, which is represented as a <a href="https://msdn.microsoft.com/en-us/library/dd942554.aspx" target="_blank">LengthPrefixedAnsiString</a>. In our case, the first four bytes represent the length of the string, which is `11` (`0x0B` in hexadecimal). The class name is `Equation.3` (including the null terminator byte `0x00`), which specifies the usage of **Equation Editor**.

![image.png](/image/rtf-document/image%203.png)

Following the **ClassName** are the **TopicName** and **ItemName** fields. Both fields are also stored as **LengthPrefixedAnsiString** values. Since the first four bytes of each field are `0x00000000`, both strings have a length of zero. This is Microsoft's way of indicating that these string fields are empty and contain no value.

![image.png](/image/rtf-document/image%204.png)

Since the **FormatID** decoded above indicates that we are dealing with an **EmbeddedObject**, the remaining bytes (shown below) must be decoded according to Microsoft's <a href="https://msdn.microsoft.com/en-us/library/dd942053.aspx" target="_blank">EmbeddedObject</a> specification. Note that the data blob extends much further than what is shown in the image below, the screenshot only captures a small portion of it.

![image.png](/image/rtf-document/image%205.png)

According to the **EmbeddedObject** specification, the first four bytes represent the **NativeDataSize** field. In our case, the value is `0x00000C00`, indicating the size of the **NativeData** field in bytes.

![image.png](/image/rtf-document/image%206.png)

Parsing the **NativeData** field, we can see that it begins with the **“file signature value”** `0xD0CF11E0A1B11AE1`.   

![image.png](/image/rtf-document/image%207.png)

This **“file signature value”** identifies the data as a <a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-cfb/53989ce4-7b05-4f8d-829b-d08d6148375b" target="_blank">Compound File Binary Format (CFBF)</a> file.

![](/image/rtf-document/image%208.png)

<p align="center">
  <em>Source: <a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-cfb/05060311-bfce-4b12-874d-71fd4ce63aea" target="_blank">[MS-CFB]: Compound File Header</a></em>
</p>

If we continue parsing the **NativeData** field, we eventually find the command that gets executed once the vulnerability in **Equation Editor** has been successfully exploited.

![image.png](/image/rtf-document/image%209.png)

Again, I won’t dive too DEEP into how **Compound File Binary Format** works or its internal structures.

However, this should give you a clear picture of what is happening. The RTF document contains an embedded **OLE** **Object** (from the `\objemb` control word). By parsing the **ObjectHeader** stored in the `\objdata` control word, we can determine that we are dealing with an `Equation.3` **EmbeddedObject**. As we continue parsing the **EmbeddedObject**, we discover a file signature indicating that it contains a **Compound File Binary Format (CFBF)** file, which contains the malicious payload that gets executed once the vulnerability in **Equation Editor** is successfully exploited.

![](/image/rtf-document/image%2010.png)

<p align="center">
  <em>Source: <a href="https://en.wikipedia.org/wiki/Object_Linking_and_Embedding" target="_blank">Object Linking and Embedding - Wikipedia</a></em>
</p>

&#x21d2; This tells us that OLE objects are stored using the **Compound File Binary Format (CFBF)**, which is based on the <a href="https://en.wikipedia.org/wiki/File_Allocation_Table" target="_blank">File Allocation Table (FAT)</a> concept.

What catch my attention is the following statement from Wikipedia:

> OLE allows an editing application to export part of a document to another editing application and then import it with additional content.
> 

This statement is quite interesting. It tells us that an OLE object can be embedded into a completely different application. The word **"different"** is important here. It means that the object can be created by one application and later opened, displayed, or edited by another application that understands the object type.

But this raises an important question.

If an OLE object is embedded inside a document and later opened on **“a different computer”**, how does Windows know to handle that object?

The answer lies in the **Component Object Model (COM)**.

![](/image/rtf-document/image%2011.png)

<p align="center">
  <em>Source: <a href="https://en.wikipedia.org/wiki/Component_Object_Model" target="_blank">Component Object Model - Wikipedia</a></em>
</p>

With the help of **COM**, OLE objects can work across different applications and even on different computers, as long as the appropriate software is installed 😉

To summarize, if I want to embed an Excel spreadsheet into a Word document, I should create an OLE object that either references the spreadsheet or embeds it directly inside the Word document. This OLE object contains **“instructions”** that only the **COM Interface** for the Word know which application is responsible for handling the object.

In reality, Word does not understand the internal structure of an Excel spreadsheet. Instead, Word relies on **COM Interface** to locate and communicate with the application responsible for handling that object. **COM Interface** takes care of the heavy things behind the scenes, allowing Word to display, edit, or interact with the embedded Excel spreadsheet.

Here, I have a diagram that will make things easier to understand.

![image.png](/image/rtf-document/image%2012.png)

In the case of `CVE-2017-11882`, Word encounters an OLE object stored inside the `\objdata` field. Earlier, we have discovered the **ClassName** field is `Equation.3`, so Word uses **COM** to locate the application responsible for handling `Equation.3` object, which is **Equation Editor**.

Once **Equation Editor** is launched and begins processing the embedded object data, the vulnerability is triggered, eventually leading to the execution of the attacker's payload.

Before moving on, keep this in your mind.

## Smashing the Stack

Because `CVE-2017-11882` abuses a buffer overflow in the **FontName** field to execute its payload, I want to first arm you guys with some basic knowledge about <a href="https://en.wikipedia.org/wiki/Stack_buffer_overflow" target="_blank">stack buffer overflow</a>.

But don't worry, we'll get to the actual `CVE-2017-11882` exploit shortly after. I just want you guys to have a bird's-eye view of what's happening before we dive into the technical details!

Here, I have a simple C program to demonstrate a buffer overflow vulnerability.

```c
#include <stdio.h>
#include <string.h>

void copy_str(char* arg1) {
    char buf[5];
    strcpy(buf, arg1);
}

int main() {
    char* myStr = "HAHA";
    copy_str(myStr);

    return 0;
}
```

The program's flow is fairly simple. It copies the value of the string `myStr` into the local variable `buf` inside the `copy_str` function. The string `myStr` has a length of 5 bytes (including the null terminator `\0`), and the local buffer `buf` also has a size of 5 bytes. As a result, no buffer overflow occurs.

The following image shows what the stack frame of the `copy_str` function looks like after the string has been copied successfully.

![image.png](/image/rtf-document/image%2013.png)

However, the function `strcpy()` has a well-known vulnerability where it keeps copying bytes until it encounters a null terminator (`\0`).

![](/image/rtf-document/image%2014.png)

<p align="center">
  <em>Source: <a href="https://cplusplus.com/reference/cstring/strcpy/" target="_blank">strcpy - cstring</a></em>
</p>

&#x21d2; This means that if the source string is larger than the destination buffer, `strcpy()` will continue writing beyond the boundaries of the destination buffer. As a result, it can overwrite important values on the stack, such as the **Old EBP** (`ebp`) and the **Return Address** (`eip`).

In the example above, imagine that `myStr` contains the following value:

```
HAHAxxxxx01020304
```

The dangerous thing about this string is that the extra bytes (`01020304`) can overwrite the **Return Address**. If an attacker replaces the **Return Address** with a valid address, such as the address of `WinExec()`, then when the `copy_str()` function returns, the program will jump to `WinExec()` instead of returning to the original code. In other words, the attacker can change where the program goes next and force it to execute code of their choice.

The following image demonstrates this scenario.

![image.png](/image/rtf-document/image%2015.png)

## Debugging Challenges

One of the biggest hurdles when analyzing this CVE is that the vulnerable process, `EQNEDT32.EXE`, is launched as a child process of Microsoft Word. Standard **F9** debugging often fails because the process spawns and terminates almost immediately.

To help you guys visualize the challenge, I use a tool called <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/procmon" target="_blank">Procmon</a> with the following filter.

![image.png](/image/rtf-document/image%2030.png)

When opening the malicious RTF file, the following processes are spawned.

![image.png](/image/rtf-document/314683ba-c6bd-4dba-9783-a5e478193849.png)

Checking the **Process Tree** inside **Procmon**, it is clear that `EQNEDT32.EXE` terminates exceptionally fast.

![image.png](/image/rtf-document/9aadf576-57ba-4db0-9802-02c710565a57.png)

This creates a pretty annoying problem. By the time we notice the process and try to attach a debugger to it, the process is already gone 😅

Haha, don't worry! I have two solutions for this problem, and they are amazing. Honestly, even I was surprised when I discovered them xD

## Solution #1 - GFlags

I learn this trick from `@cybercdh`’s Youtube video. You guys can check out his amazing video from <a href="https://www.youtube.com/watch?v=aBWAHxpXHEk&t=839s" target="_blank">here</a>.

Basically, **GFlags (Image File Execution Options)** is a tool that allows us to force a debugger to attach the moment `EQNEDT32.EXE` starts, ensuring that we never miss the overflow event.

You can get it by installing the **WDK**. The following link will guide you through the installation process.

&#x2192; <a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/download-the-wdk#download-icon-for-wdk-step-3-install-the-wdk" target="_blank">Download the Windows Driver Kit (WDK)</a>

Here are the paths to `gflags.exe`:

```bash
 -------------
| x64 Version |
 -------------
 
C:\Program Files (x86)\Windows Kits\10\Debuggers\x64

 -------------
| x86 Version |
 -------------
 
C:\Program Files (x86)\Windows Kits\10\Debuggers\x86
```

Since `EQNEDT32.EXE` is a **32-bit** binary, let's use the **32-bit** version of **GFlags**.

![image.png](/image/rtf-document/image%2031.png)

Simply go to **Image File** &#x2192; **Image** (remember to press **TAB**) &#x2192; **Debugger** (enter the path to **x32dbg**) &#x2192; **Apply**.

![image.png](/image/rtf-document/image%2032.png)

Now, opening the RTF file should launch **x32dbg**. But what is this error!!?? 

![image.png](/image/rtf-document/6f7e4405-4d92-465e-93bb-af5dc15a8113.png)

After crushing my head over this problem for nearly a week, I have finally figured out that the error was caused by an incompatible version of **x32dbg**.

Here is the version I was originally using. It was built on **8/20/2025**.

![image.png](/image/rtf-document/image%2033.png)

Since this build is fairly new, my guess is that there is some issue with how it handles the attach process.

Here is my alternative:

&#x2192; <a href="https://x64dbg.en.uptodown.com/windows/download/73124393" target="_blank">Download x64dbg 2022-06-15 for Windows</a>

After updating the path inside the **Debugger** field of **GFlags** to point to this version of **x32dbg**, everything works perfectly.

![image.png](/image/rtf-document/image%2034.png)

## Solution #2 - Registry

In this section, the key is to use **Registry** and configure it to start and attach the debugger **x32dbg** as soon as the process `EQNEDT32.EXE` starts.

First of all, simply press **Windows + R** and type **"regedit"**, as shown in the image below.

![image.png](/image/rtf-document/image%2035.png)

Navigate to the following registry key:

```
HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options
```

If you have successfully completed all the steps in **"Solution #1 - GFlags"**, you should see a folder named `eqnedt32.exe`.

![image.png](/image/rtf-document/image%2036.png)

Navigate to the `eqnedt32.exe` folder, and you should see a string value named `Debugger` with the following data.

![image.png](/image/rtf-document/image%2037.png)

&#x21d2; This means that if you have successfully configured **x32dbg** through **GFlags**, it will automatically create and set the value of the `Debugger` string in the registry to your debugger of choice. As a result, **x32dbg** will launch automatically whenever `EQNEDT32.EXE` is executed.

&#x21d2; This also means that you can freely modify the value of this registry to point to any debugger that you prefer.

Opening the RTF file results in the execution of **x32dbg** as soon as the process `EQNEDT32.EXE` starts.

![image.png](/image/rtf-document/93834515-b136-4795-b888-371f7e3d80b6.png)

## Overflowing the Font

The function highlighted below is the one responsible for the overflow.

![image.png](/image/rtf-document/image%2038.png)

The first argument pushed to the function `sub_41160F` is actually the payload.

![image.png](/image/rtf-document/image%2039.png)

![image.png](/image/rtf-document/image%2040.png)

Inside `sub_41160F`, the instruction at address `0x00411658` is responsible for copying the payload into the **FONT Record**.

![image.png](/image/rtf-document/image%2041.png)

But what do these assembly instructions actually mean?

```nasm
shr ecx, 2
rep movsd
```

It turns out that these instructions are used to copy data from `esi` to `edi`.

![](/image/rtf-document/image%2042.png)

<p align="center">
  <em>Source: <a href="https://stackoverflow.com/questions/27804852/assembly-rep-movs-mechanism" target="_blank">x86 - Assembly: REP MOVS mechanism - Stack Overflow</a></em>
</p>

The instruction `shr ecx, 2` divides the value in `ecx` by 4. After that, `rep movsd` starts copying data from `esi` (source) to `edi` (destination) 4 bytes at a time, repeating the process `ecx` times. 

In short, these two instructions are doing the dirty work of copying the payload into the destination buffer.

Let’s explain things a little bit inside **x32dbg**.

After running the instruction `shr ecx, 2`, the value inside `ecx` has become `0xC` or `12` in decimal format. 

![image.png](/image/rtf-document/image%2043.png)

Also, we know that at this exact moment, the destination buffer `edi` is still empty (or contains trash values), and the source buffer `esi` should hold the payload.

![image.png](/image/rtf-document/image%2044.png)

![image.png](/image/rtf-document/image%2045.png)

![image.png](/image/rtf-document/image%2046.png)

Some of you might wonder what is the original size of the destination buffer `edi`. The answer lies directly in the assembly instruction below.

```nasm
lea edi,dword ptr ss:[ebp-28]           
```

As you guys already know, when the register `ebp` is subtracted by a hexadecimal value, it always means that it is referencing a local variable inside the function. In this case, it is a local buffer (later stored in `edi`) with a size of `0x28`, which is `40` in decimal format.

So, when the instruction `rep movsd` is executed, the destination buffer `edi` will be filled with the attacker payload from the source buffer `esi`. This process copies `48` bytes into the destination buffer `edi`. 

Some might ask how do I know the copied size is `48` bytes. Haha, the answer lies directly in the instruction `shr ecx, 2`, which divides the value inside `ecx` by 4. Since the current value of `ecx` is `0xC`, I just need to do some basic math to get the overflow size.

![image.png](/image/rtf-document/image%2047.png)

As I already showed you guys how a stack buffer overflow works in the section **"Smashing the Stack"**, this overflow of `48` bytes will overwrite the **Old EBP** and the **Return Address**.

Here is the destination buffer `edi` after executing the instruction `rep movsd`.

![image.png](/image/rtf-document/image%2048.png)

When I navigate my mouse to the last 4 bytes of data in the hex blob above, it shows the address of function `WinExec()`.

![image.png](/image/rtf-document/image%2049.png)

This means that when the overflowed function `sub_41160F` returns, it jumps to the address of `WinExec()`.

![image.png](/image/rtf-document/image%2050.png)

As you can see on the stack when **x32dbg** breaks at the instruction `ret`, the value is `0x00430C12`, which is the address of `WinExec()`.

Press **F8**, and we jump to the function `WinExec()`.

![image.png](/image/rtf-document/image%2051.png)

Based on the **MSDN Documentation**, `WinExec()` takes two arguments.

![](/image/rtf-document/image%2052.png)

<p align="center">
  <em>Source: <a href="https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-winexec#syntax" target="_blank">WinExec function (winbase.h) - Win32 apps</a></em>
</p>

The first argument is the one I am interested in the most. This is the command line that the application will execute.

![](/image/rtf-document/image%2053.png)

<p align="center">
  <em>Source: <a href="https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-winexec#parameters" target="_blank">WinExec function (winbase.h) - Win32 apps</a></em>
</p>

Back to our **x32dbg** flow, the attacker hijacks the program's execution flow by altering the **Return Address**, causing it to jump directly to the `WinExec()` function. The data for `lpCmdLine` has already been pushed onto the stack beforehand and is located at address `0x0019F188`.

![image.png](/image/rtf-document/image%2054.png)

This is the value of `lpCmdLine`.

![image.png](/image/rtf-document/image%2055.png)

After executing `WinExec()`, it spawns `cscript` to run the file `license.js` inside the `%tmp%` folder.

![image.png](/image/rtf-document/image%2056.png)
